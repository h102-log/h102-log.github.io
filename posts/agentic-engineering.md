## 에이전트에게도 개발 환경이 필요하다

AI에게 코드를 짜게 하는 일은 이제 익숙합니다. 하지만 대화창에 붙여넣고 결과를 복사하는 방식으로는 프로덕션에 올릴 만한 품질과 신뢰성을 얻기 어렵습니다.

**에이전틱 엔지니어링(Agentic Engineering)** 은 여기서 한 단계 더 나아갑니다. 단순히 AI에게 코딩을 시키는 게 아니라, **AI 에이전트가 프로 수준의 품질을 유지하며 자율적으로 일할 수 있도록 공학적인 환경과 규율을 설계**하는 개발 패러다임입니다.

이 개념은 에이전트를 **완전히 새로 들어온 신입 개발자(사람)** 라고 생각하면 훨씬 쉽게 이해됩니다. 신입에게 좋은 성과를 기대하려면, 온보딩 문서와 노트북과 테스트 서버를 갖춘 **제대로 된 근무 환경**을 마련해 줘야 합니다. 그 환경을 이루는 네 개의 기둥이 바로 **Context**, **Agentic Validation**, **Agentic Tooling**, **Agentic Codebases** 입니다.

---

## 기존 방식: 메모장에 코드 받아쓰기

에이전틱 엔지니어링이 왜 필요한지는, 그 이전 방식과 비교하면 분명해집니다. 초기의 AI 코딩, 이른바 **바이브 코딩(vibe coding)** 은 신입 개발자를 **"방 한 칸에 가둬두고, 컴퓨터도 안 주고, 메모장으로만 일 시키는 방식"** 에 가까웠습니다.

**1) Context 부족**

신입에게 파일 하나만 툭 던지고 "대충 눈치껏 고쳐봐"라고 합니다. 회사의 코딩 스타일도, 전체 폴더 구조도, 쓰는 라이브러리도 알려주지 않습니다. 맥락을 모르니 엉뚱한 코드를 짜오기 일쑤였습니다.

**2) Tooling 없음**

노트북도 없이 종이와 펜만 주고 "코드를 적어서 제출해"라고 합니다. 신입은 자기 코드가 실행은 되는지, 오타는 없는지 **직접 돌려볼 수가 없습니다.** 오직 머릿속 시뮬레이션으로만 코딩해야 했습니다.

**3) 수동 에러 피드백**

신입이 종이에 적어 주면, 사수가 그걸 받아 직접 타이핑하고 실행해 봅니다. 에러가 나면 에러 메시지를 다시 종이에 적어 전달합니다. **사수가 일일이 손으로 에러를 먹여** 키워야 하는 구조였습니다.

**4) 꼬인 코드베이스**

집(코드베이스) 구조가 너무 복잡해서, 신입이 방 하나 고치려다 벽을 허물었더니 집 전체가 무너집니다. AI가 변경의 **사이드 이펙트(side effect)** 를 감당하지 못했던 것입니다.

이 비효율을 걷어내자며 나온 것이 에이전틱 엔지니어링입니다. 흐름으로 보면 이렇게 바뀝니다.

```text
[기존 방식] ─▶ 사람(AI)에게 단편 질문 던지고 답변 복붙 (수동)
[에이전틱]  ─▶ 사람(AI)에게 규정집 + 노트북 주고 테스트 서버에서 직접 실행
```

## 1) Context — 목표와 규칙 알려주기

**Context(문맥)** 는 에이전트가 지금 **어디서, 무엇을 위해, 어떤 규칙으로** 코드를 짜는지 이해하게 만드는 기반입니다. 신입이 첫날 받는 **온보딩 문서이자 사내 규정집**에 해당합니다.

방법은 명세 파일을 프로젝트에 두는 것입니다. `CLAUDE.md` 나 `.cursorrules` 같은 파일에 팀의 코딩 컨벤션과 아키텍처 가이드라인을 담아 에이전트에게 주입합니다.

```markdown
# 프로젝트 규칙

- 들여쓰기는 스페이스 2칸을 사용한다
- 상태 관리는 Redux Toolkit을 사용한다
- 모든 API 응답은 표준 에러 포맷을 따른다
```

또 하나의 핵심은 **작업 전에 계획(Plan)을 먼저 제출**하게 만드는 것입니다. 에이전트에게 전체 설계를 요약하고 접근 방법을 내놓게 한 뒤 착수시키면, 엉뚱한 방향으로 코드를 쏟아내는 **환각(hallucination)** 을 크게 줄일 수 있습니다.

---

## 2) Agentic Validation — 결과를 검증하기

**Agentic Validation(검증)** 은 에이전트가 만든 결과물이 프로덕션에 올려도 될 만큼 안전한지 걸러내는 **품질 관리(QC) 하네스(harness)** 입니다. 신입이 코드를 낼 때마다 자동으로 돌아가는 **검증 시스템**인 셈입니다.

에이전트가 코드를 짜면 곧바로 **단위 테스트(unit test)**, 통합 테스트, 정적 분석(linting), 보안 스캔을 실행하게 만듭니다.

```bash
npm run lint
npm test
npm run build
```

흥미로운 점은 **구현 에이전트와 검증 에이전트를 분리**하는 방식입니다. AI가 짠 코드를 **다른 AI가 비판적으로 교차 검증**하게 만드는 멀티 에이전트 파이프라인입니다. 옆자리 신입에게 "내 코드 좀 리뷰해줘"라고 시키는 것과 같습니다.

> 검증은 사람이 매번 손으로 하던 일을 자동화한 것입니다. 사수가 종이를 받아 타이핑하던 자리를, 테스트 스위트와 리뷰 에이전트가 대신합니다.

---

## 3) Agentic Tooling — 손과 발 달아주기

**Agentic Tooling(툴링)** 은 에이전트가 텍스트만 주고받는 데 그치지 않고, 개발 환경에서 실제로 **행동**하도록 손과 발을 달아주는 기술입니다. 신입에게 지급하는 **개발용 노트북**에 해당합니다.

에이전트가 터미널을 열고, 파일을 읽고 쓰고, 데이터베이스에 쿼리를 날리고, API를 호출할 수 있도록 인터페이스를 연결합니다. 최근에는 **MCP(Model Context Protocol)** 서버나 여러 Agent SDK가 이 역할을 맡습니다.

이 덕분에 에이전트는 파일을 직접 수정하고, 빌드 명령을 입력하고, 에러 로그를 확인해 **자가 수정(self-correction)** 까지 처리합니다. 종이에 코드를 적는 게 아니라, 노트북을 열고 실행해 보며 스스로 고치는 것입니다.

> 주의: 파일 쓰기·삭제나 셸 실행처럼 되돌리기 어려운 도구 권한은 신중히 부여해야 합니다. 강력한 손발은 잘못 쓰이면 그만큼 위험합니다.

---

## 4) Agentic Codebases — 일하기 좋은 무대

**Agentic Codebases** 는 AI 에이전트가 읽고, 이해하고, 수정하기 좋도록 소스 코드와 저장소 구조 자체를 최적화하는 접근입니다. 신입이 헤매지 않도록 정돈해 둔 **일하기 좋은 사무실 구조**에 해당합니다.

핵심은 모듈 간 **결합도를 낮추고(decoupling)** 경계를 명확히 하는 것입니다. 에이전트가 특정 기능을 수정할 때 엉뚱한 파일까지 건드려 스파게티 코드를 만들지 않도록 **격리(sandbox)** 합니다.

- 모듈을 잘 캡슐화해 한 부분만 안심하고 수정하도록 만든다
- 코드마다 명확한 문서와 주석을 유지한다
- 에이전트가 이해하기 쉬운 단순하고 명확한 아키텍처 패턴을 유지한다

집 구조가 직관적일수록 신입은 길을 잃지 않고, 벽 하나를 고치다 집 전체를 무너뜨릴 일도 없습니다.

---

## 정리

네 기둥을 신입 개발자 비유로 한눈에 보면 다음과 같습니다.

| 기둥 | 사람 비유 | 하는 일 |
| --- | --- | --- |
| Context | 온보딩·사내 규정집 | 목표·범위·컨벤션 주입, Plan 우선 |
| Agentic Validation | 자동 검증 시스템 | 테스트·린트·보안·교차 리뷰 |
| Agentic Tooling | 개발용 노트북 지급 | 터미널·파일·API 접근, 자가 수정 |
| Agentic Codebases | 일하기 좋은 사무실 | 모듈 격리·문서화·명확한 구조 |

기존에는 AI라는 **뇌**만 빌려 나머지 수동 작업을 전부 사람이 처리했다면, 에이전틱 엔지니어링은 AI가 **독립된 개발자처럼 일하도록 환경 자체를 깔아주는** 일입니다.

핵심은 **"AI에게 뇌만 빌리지 말고, 사람처럼 일할 환경(규정집·노트북·검증기·사무실)을 깔아주라"** 는 것입니다. Context로 목표를 정하고, Tooling으로 실행력을 주며, Codebases라는 안전한 무대 위에서, Validation으로 완벽하게 검증해 내는 것. 이것이 에이전틱 엔지니어링의 구조입니다.
