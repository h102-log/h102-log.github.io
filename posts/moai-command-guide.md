## /moai 하나로 끝내는 개발 워크플로우

MoAI-ADK는 Claude Code 위에서 동작하는 오케스트레이션 프레임워크입니다.
모든 작업의 진입점은 `/moai` 하나이며, 뒤에 붙는 **서브커맨드**에 따라 적절한 전문 에이전트로 작업이 위임됩니다.

서브커맨드는 크게 세 부류로 나뉩니다.

| 구분 | 서브커맨드 | 역할 |
| --- | --- | --- |
| 워크플로우 | `plan`, `run`, `sync` | SPEC 기반 3단계 개발 |
| 유틸리티 | `fix`, `loop`, `project` | 빠른 수정·반복·초기화 |
| 피드백 | `feedback` | 버그 리포트·개선 제안 |

그리고 서브커맨드 없이 자연어만 입력하면, `plan → run → sync` 전체 파이프라인이 **자동으로** 실행됩니다.

하나씩 살펴보겠습니다.

---

## 1) /moai plan — 명세부터 시작하기

```
/moai plan "사용자가 이메일로 로그인할 수 있어야 한다"
```

`plan`은 **SPEC 문서**를 만드는 단계입니다. `manager-spec` 에이전트가 요구사항을 EARS 형식으로 정리해 `.moai/specs/SPEC-XXX/spec.md`에 저장합니다.

EARS는 "When [조건], the system shall [동작]" 같은 구조화된 요구사항 문법으로, 모호함을 줄이고 인수 조건(acceptance criteria)을 명확히 합니다.

**이럴 때 사용하세요.**

- 기능이 어느 정도 규모가 있어서 바로 코딩하기 전에 요구사항을 정리하고 싶을 때
- 인수 조건과 기술적 접근 방식을 문서로 남겨 팀과 공유해야 할 때
- 나중에 구현(`run`)과 문서화(`sync`)까지 이어갈 계획일 때

> `plan` 완료 후에는 `/clear`로 컨텍스트를 비우는 것이 권장됩니다.
> 명세 작성에 쓴 토큰을 정리해야 이어지는 구현 단계에서 더 큰 작업 공간을 확보할 수 있습니다.

---

## 2) /moai run — 명세대로 구현하기

```
/moai run SPEC-AUTH-001
```

`plan`으로 만든 SPEC ID를 넘기면, `manager-ddd` 에이전트가 **DDD(Domain-Driven Development)** 방식으로 구현합니다.

DDD는 **ANALYZE → PRESERVE → IMPROVE** 사이클로 동작합니다.

1. **ANALYZE** — 기존 코드와 의존성, 도메인 경계를 파악
2. **PRESERVE** — 현재 동작을 고정하는 특성 테스트(characterization test) 작성
3. **IMPROVE** — 작은 단위로 변경하며 매번 테스트로 검증

즉, 기존 동작을 깨지 않으면서 점진적으로 기능을 더하는 방식입니다. 완료 기준은 SPEC 요구사항 충족, 테스트 통과, 85% 이상 커버리지, TRUST 5 품질 게이트 통과입니다.

**이럴 때 사용하세요.**

- 이미 `plan`으로 SPEC을 확정했고 그대로 구현에 들어갈 때
- 레거시 코드를 동작 보존하면서 안전하게 개선할 때

---

## 3) /moai sync — 문서화와 PR 자동화

```
/moai sync SPEC-AUTH-001
```

구현이 끝나면 `sync`가 마무리를 담당합니다. `manager-docs` 에이전트가 API 문서, README 업데이트, CHANGELOG 항목을 만들고 풀 리퀘스트까지 준비합니다.

**이럴 때 사용하세요.**

- 구현과 테스트가 끝나 배포 전 문서를 정리할 때
- PR을 자동으로 생성해 리뷰 단계로 넘기고 싶을 때

여기까지가 `plan → run → sync`로 이어지는 표준 3단계 워크플로우입니다.

```
/moai plan "기능 설명"     ← SPEC 작성
        ↓ (/clear)
/moai run SPEC-XXX          ← 구현
        ↓
/moai sync SPEC-XXX         ← 문서화 + PR
```

---

## 4) /moai (자연어) — 전체 파이프라인 자동 실행

```
/moai 인증 시스템 만들어줘
```

서브커맨드 없이 자연어로 요청하면, 위 3단계가 **한 번에** 자동 실행됩니다. MoAI가 요청을 분석해 SPEC 작성부터 구현, 문서화까지 알아서 진행합니다.

**이럴 때 사용하세요.**

- 단계를 직접 끊어가며 통제할 필요 없이 한 번에 처리하고 싶을 때
- 비교적 명확한 기능을 빠르게 끝에서 끝까지 완성하고 싶을 때

> 요청이 모호하면 MoAI가 곧바로 실행하지 않고, `AskUserQuestion`으로 의도를 먼저 확인합니다.
> "더 좋게 만들어줘" 같은 막연한 지시보다 대상과 범위를 구체적으로 적는 것이 좋습니다.

---

## 5) /moai fix — 한 방 자동 수정

```
/moai fix
```

`fix`는 식별된 문제를 **한 번에** 고치는 원샷 명령입니다. `expert-debug`, `expert-refactoring` 에이전트가 투입됩니다.

**이럴 때 사용하세요.**

- 에러 메시지나 린트 경고가 명확해서 빠르게 한 번 고치면 되는 상황
- 전체 워크플로우를 돌릴 만큼 크지 않은 단발성 버그

---

## 6) /moai loop — 끝까지 반복 수정

```
/moai loop
```

`loop`는 문제가 모두 해결될 때까지 **자동으로 반복**합니다. `fix`가 한 번이라면, `loop`는 LSP 진단과 테스트를 기준으로 에러가 0이 될 때까지 수정-검증을 되풀이합니다.

**이럴 때 사용하세요.**

- 에러가 여러 개 얽혀 있어 한 번의 수정으로 끝나지 않을 때
- 타입 에러·린트·테스트 실패를 한꺼번에 깔끔히 정리하고 싶을 때

| 명령 | 동작 | 적합한 상황 |
| --- | --- | --- |
| `fix` | 1회 수정 | 명확한 단발 버그 |
| `loop` | 통과할 때까지 반복 | 다수의 얽힌 에러 |

---

## 7) /moai project — 프로젝트 초기화

```
/moai project
```

`manager-project` 에이전트가 새 MoAI 프로젝트의 설정과 구조를 잡아줍니다. `.moai` 디렉터리 구성, 템플릿, 언어 설정 등을 초기화합니다.

**이럴 때 사용하세요.**

- MoAI-ADK를 처음 도입하는 새 프로젝트를 셋업할 때
- 프로젝트 구조나 설정을 표준 템플릿으로 정리하고 싶을 때

---

## 8) /moai feedback — 피드백 보내기

```
/moai feedback
```

버그 리포트나 개선 제안을 MoAI-ADK 저장소에 GitHub 이슈로 등록합니다.

**이럴 때 사용하세요.**

- MoAI 사용 중 버그를 발견했을 때
- 새로운 기능이나 개선 아이디어를 제안하고 싶을 때

---

## 정리

| 서브커맨드 | 핵심 역할 | 담당 에이전트 |
| --- | --- | --- |
| `plan` | SPEC 문서 작성 | manager-spec |
| `run` | DDD 구현 | manager-ddd |
| `sync` | 문서화·PR | manager-docs |
| (자연어) | plan→run→sync 자동 | 복수 |
| `fix` | 1회 자동 수정 | expert-debug |
| `loop` | 반복 자동 수정 | expert-debug 외 |
| `project` | 프로젝트 초기화 | manager-project |
| `feedback` | 이슈 등록 | - |

처음 쓴다면 **자연어 입력**으로 가볍게 시작하고, 규모 있는 기능은 `plan → run → sync`로 단계를 끊어 통제하는 방식을 추천합니다.
버그 대응은 단발이면 `fix`, 여러 개면 `loop`로 구분하면 됩니다.
